# 10 · L0.6：教一个 124M 模型用工具

> [← end-to-end trace](09-end-to-end-trace.md) ｜ 代码：[`00c_agent_sft/`](https://github.com/fxp/LLM-from-query-to-result/tree/main/00c_agent_sft) + [`02_agent/agent_loop.py`](https://github.com/fxp/LLM-from-query-to-result/blob/main/02_agent/agent_loop.py)

这本书写到一半我意识到一件事：从 L0 到 L5 我把"模型怎么诞生 + 怎么 serve"讲全了，**但用户问"agent 在哪？"是个 fair 的问题**。

[L2](04-L2-chat-client.md) 那一篇我专门讲过为什么删掉了原来的 agent loop——124M base model 做不了 tool use，硬撑那段代码就是"我们用 Claude API"伪装成"我们写了 agent"。

但现在，**有了 L0 base + L0.5 instruction SFT + L3 KV cache infra**，加一层 L0.6 教模型 ReAct 格式，agent 就能真的工作。**33 秒 SFT，258 条合成 trace**。

这一篇讲那 33 秒里到底发生了什么，以及为什么这是 agent 真正的核心机制。

## 想要的输出

最终要的是这种交互：

```
👤 What is 1234 plus 5678?
💭 I need to compute 1234 + 5678.
🔧 calc(1234 + 5678)
   ↳ 6912
🤖 6912.
```

注意 **1234+5678 不在 SFT 训练数据里**——SFT 只见过 1-99 范围内的随机加法。但 model 在 inference 时能**正确生成 `calc(1234 + 5678)`** 把任意数字传给工具。**model 的能力扩展靠工具，不靠记忆**。

这是 agent 的本质。一个 124M model 学会了"我不会算，但我知道有个 calc 工具"——这件事比 model 凭脑补答 "70" 更有用。

## 格式：ReAct

[ReAct paper (Yao et al. 2022)](https://arxiv.org/abs/2210.03629) 的格式：

```
Q: <question>
THOUGHT: <reasoning>
ACTION: tool_name(args)
OBSERVATION: <real tool output>     ← 来自工具，不是模型
THOUGHT: ... (optional)
ACTION: ...
OBSERVATION: ...
ANSWER: <final><|endoftext|>
```

**关键**：`OBSERVATION:` 那一段内容**不能让 model 自己编**。它必须等模型 emit `OBSERVATION:` 这个 prefix（作为 stop signal），然后 agent loop 拦截、调用真实工具、把工具输出注入回 prompt，让 model 继续生成。

如果 model 自己 emit 了 OBSERVATION 内容，那是**幻觉**——它在凭脑补编工具结果。这是 agent 跟 chat completion 最关键的区别：**OBSERVATION 由代码控制，不由模型控制**。

## SFT 数据：程序生成

158 条 lookup + 85 条 calc + 14 条 paraphrase = 258 条 trace。我用 [`build_data.py`](https://github.com/fxp/LLM-from-query-to-result/blob/main/00c_agent_sft/build_data.py) 程序生成：

```python
def trace_calc(question, expr, answer_template):
    obs = tools.calc(expr)   # 真调工具产生 obs，保证训练时 obs 跟 runtime obs 完全一致
    return {
        "q": question,
        "trace": [{
            "thought": f"I need to compute {expr}.",
            "action": f"calc({expr})",
            "observation": obs,
        }],
        "answer": answer_template.format(obs=obs),
    }

# 例子
trace_calc("What is 23 plus 47?", "23 + 47", "{obs}.")
# → {"q": "...", "trace": [{"thought": "I need to compute 23 + 47.",
#                           "action": "calc(23 + 47)",
#                           "observation": "70"}],
#    "answer": "70."}
```

**为什么程序生成而不是手写**：
1. 真实 instruction-tuning 数据集都是合成或半合成（Alpaca 是 GPT-3.5 生成的；OpenOrca 是 GPT-4）。
2. 训练数据里 `observation` 由真工具算出来，跟 runtime 时工具返回的字符串**完全一致**。手写容易写错（"70.0" vs "70"）。
3. 改 KB 自动重生成数据，零维护成本。

## Loss masking — 我搞错过的关键细节

每条 trace 转成 `(input_ids, target_ids)` 时要决定每个 token 是"学"还是"mask"：

```python
parts.append((f"Q: {q}\n", False))                       # 用户给的，不学
parts.append((f"THOUGHT: {thought}\n", True))            # model 自己生成
parts.append((f"ACTION: {action}\n", True))              # model 自己生成
parts.append(("OBSERVATION:", True))                     # model 自己生成 ← !!
parts.append((f" {observation}\n", False))               # 工具给的，不学
parts.append((f"ANSWER: {answer}", True))                # model 自己生成
```

注意 `OBSERVATION:` 这 prefix **被分开**——prefix 自身（"OBSERVATION:" 这 5 个字符的 token）是"学"的，后面的实际内容（` Paris\n`）才 mask。

**第一版我搞错了**——把整行 `f"OBSERVATION: {observation}\n"` 都 mask 了。结果 model 训完会循环：

```
THOUGHT: I should look up capital of France.
ACTION: lookup(capital of France)
ACTION: lookup(capital of France)        ← 第二个 ACTION！
ACTION: lookup(capital of France)
ACTION: lookup(...)
...
```

为什么循环？因为：
- `ACTION:` 行末的 `\n` 是"学"的，model 学到 emit 它
- 接下来该 emit 什么？训练数据里下一段是 `OBSERVATION: ...\n`，但**全部 mask 了**
- backprop 没经过这段，model 没学到 "\\n" 后跟 "OBSERVATION:"
- 它默认按 prior 续写——而 prior 是它之前学得最熟的 "ACTION:"

修复非常小：把 `"OBSERVATION:"` 这 prefix 单独抽出来 LEARN。**1 行代码改动**，model 立刻学会发 stop signal。

这是 agent SFT 数据 masking 最容易犯的错误，也是为什么 SFT 数据格式必须想清楚再写。

## 训练（33.5 秒）

```
loaded base: 124.44M params
agent traces: 258  avg len=48.2 tokens

Agent SFT for 20 epochs × 33 steps = 660 updates
  batch_size=8  lr=5e-05  warmup=30

epoch   0/20 | avg loss 2.293 |  2.1s
epoch   2/20 | avg loss 0.001 |  5.4s    ← 第 2 轮就基本零 loss
epoch   4/20 | avg loss 0.000 |  8.8s
epoch   8/20 | avg loss 0.000 | 15.5s
epoch  19/20 | avg loss 0.000 | 33.5s
```

为什么收敛快：258 traces × ~25 learn-tokens × 20 epochs = ~130K target tokens 反向传播。124M 模型的 FFN 容量绰绰有余记住这点东西。这跟 L0.5 path B 一样的剧本——**大 base 的 SFT 看起来像背诵，但学的是格式 + 行为，不是知识**。

## Agent loop（运行时）

`agent_loop.py` 的核心 ~20 行：

```python
def run_agent(query):
    prompt = f"Q: {query}\n"
    for step in range(MAX_STEPS):
        # 流式 generate from L3，遇到 "OBSERVATION:" 就停
        chunk, stop = generate_until_stop(
            prompt,
            stop_strings=["OBSERVATION:", "<|endoftext|>"],
            max_tokens=100,
        )
        # 把 chunk 里的 THOUGHT/ACTION/ANSWER 当 event 抛给 L1 (UI 渲染)
        for event in parse_events(chunk): yield event

        if stop == "<|endoftext|>":
            yield {"type": "done"}; return

        # stop == "OBSERVATION:" → model 等工具结果。找最近的 ACTION，调工具。
        action = last_action(chunk)              # e.g. "calc(23 + 47)"
        obs = tools.call(action)                  # → "70"
        yield {"type": "observation", "v": obs}

        # 把真实 observation 注入 prompt，继续循环
        prompt = prompt + chunk + f"OBSERVATION: {obs}\n"
```

只有这一段是"agent"的核心代码。所有的"看起来像 ChatGPT 的工具调用"都是这个循环：
1. **Generate**: 让 model emit THOUGHT + ACTION + 等 OBSERVATION
2. **Parse**: 从输出里提取 `ACTION: tool(args)` 字符串
3. **Execute**: 调真工具（不是 LLM 调，是 Python 代码调）
4. **Inject**: 把真实工具输出拼回 prompt
5. **Loop**: 让 model 继续——可能再 emit THOUGHT/ACTION（多步），也可能直接 ANSWER（结束）

## 实测：what works, what doesn't

| Query | Output | 评 |
|---|---|---|
| `capital of France?` | `Paris.` ✓ | KB 命中 |
| `Who wrote Hamlet?` | `Shakespeare.` ✓ | KB 命中 |
| `chemical symbol of gold?` | `Au.` ✓ | KB 命中 |
| `What is 8 times 7?` | `56.` ✓ | calc tool 真算 |
| **`What is 1234 plus 5678?`** | **`6912.`** ✓ | **calc 算超 SFT 范围的数** |
| `What is 10 divided by 3?` | `3.3333...` ✓ | float division |
| `capital of Mongolia?` | `not found.` | KB 没有 → 诚实复述 |
| `Mona Lisa?` | model 用了 `painter of **the** Mona Lisa` → `not found` | normalize 错 |
| `How are you today?` | hallucinate `lookup(author of this paper)` | 完全 OOD |

**`1234+5678 → 6912` 是 agent 的胜利**。124M 自己绝对算不对（chat-SFT 那一版我们看过）。但带了 calc 工具，它泛化到了任意数字。

`Mongolia → not found` 也很有意思——model **不会编个城市名出来**，而是诚实复述工具返回的"not found"。这是 SFT data 没教过的行为（KB 里没 Mongolia 的训练数据），但 model 把 calc 的"工具结果是事实"思维迁移过来了。

而 `How are you today?` 完全失败——model 不会礼貌回应，只会 try lookup。这就是 124M base 的边界。把 base 换成 Llama-3-8B-Instruct 这种问题就消失了。

## 这就是 agent 真正的核心

我希望读完这篇你能带走两个 takeaway：

1. **Agent 不靠 model 更聪明，靠工具扩展能力**。124M 不会算大数，但带 calc 就行。这跟 ChatGPT 接 web_search、Cursor 接 grep+edit 是一样的本质——**外部工具是 model 的延伸**，agent loop 是协调它们的胶水。

2. **Agent SFT 跟 chat SFT 几乎一样，多了一个 masking 细节**。教 model emit `OBSERVATION:` prefix 是 stop signal，mask `<observation_content>` 不让它幻觉工具输出。除此之外训练 loop / 数据格式 / 学习率全跟 [L0.5](02-L0.5-sft.md) 一样。

ChatGPT 和 Claude 在这个 layer 上不再多神秘——它们的 tool-use 也是同样的 ReAct 风格 SFT，只是 base 更大、tools 更多、SFT 数据上百万条。**核心算法和 agent_loop.py 那 20 行一样**。

## 与 L0/L0.5 一起看

| 层 | 学的是 | 数据 | 训练时间 (5090) |
|---|---|---|---|
| L0 base pretrain | "续写英文" | 1.1 MB Shakespeare | 12 sec |
| L0.5 chat SFT | "看到 Q 答 A" | 242 Q/A 手写 | 24 sec |
| L0.6 agent SFT | "用工具算/查" | 258 ReAct traces | 33 sec |

三个一起 = **70 秒 GPU 时间**，从零造出一个能聊天 + 能调工具的 agent。代价是它只会几十种事实和初等运算——但**循环本身是真的**，scaling 是 data + compute 的事。

## 接下来想做什么

按 ROI：

1. **Multi-step trace**：现在所有 trace 都是 1 步 ACTION。加 30 条 "lookup X 然后 calc X 的 day count" 这种 2-3 步 trace，model 应该能学会多轮。
2. **更多工具**：加 `length(string)`、`reverse(string)`、`time()` 等 stateless 小工具，看 model 工具选择能不能泛化。
3. **Stateful agent**：加一个 `read_file(path)` / `write_file(path, content)` 工具，把 agent 升级成能跟文件系统交互的助手——回到这本书最早删掉的那个"做 Todo 网站"的 demo，但现在用我们自己训的 model 而非 Claude API。
4. **错误恢复**：让 model 学到"OBSERVATION 是 error/not found 时怎么处理"。
5. **DPO/RLHF**：当前的 SFT trace 都是"理想"路径，model 没见过失败 → 重试的样本。RLHF 能教这个。

每一项都是 ~100 行代码 + 几小时 GPU。
